# 骑手智能客服 - 子智能体业务划分需求分析

## 一、文档概述

### 1.1 背景

当前系统采用**单体 Agent 架构**，所有业务逻辑集中在一个 `riderAgent.ts` 中，通过一个 LLM 调用（`selectTool`）做工具选择，再由同一个 SYSTEM_PROMPT 生成回复。随着业务工具从 5 个增长到 6 个（新增天气查询），且存在大量组合意图场景（如"今天天气怎么样，能跑几单？"），单体架构的局限性日益明显。

### 1.2 目标

将单体 Agent 拆分为**主控 Agent + 多个子智能体**的层级架构，实现：

- **职责隔离**：每个子智能体专注一个业务领域，Prompt 更精准
- **并行处理**：多意图场景并行分发，缩短响应时间
- **独立演进**：新增业务只需新增子智能体，不影响已有模块
- **可测试性**：每个子智能体可独立进行单元测试和效果评估

### 1.3 适用范围

本文档覆盖：
- 业务领域分析与子智能体划分
- 主控 Agent 路由分发设计
- 各子智能体的职责、Prompt、工具、意图、典型对话场景
- 跨智能体协作流程
- 数据流与接口设计

## 二、现有架构分析

### 2.1 当前处理流程

```
用户消息
  │
  ├─ selectTool() ← LLM 调用 #1：工具选择（6选1）
  │     │
  │     ├─ 选中工具 → invokeTool() → generateReply() ← LLM 调用 #2
  │     │
  │     └─ 未选中 → 带历史上下文直接对话 ← LLM 调用 #2
  │
  └─ 保存到数据库
```

**问题清单**：

| # | 问题 | 影响 | 严重程度 |
|---|------|------|---------|
| 1 | `selectTool` 只能选一个工具 | 无法处理组合意图 | 高 |
| 2 | 所有业务共用一个 SYSTEM_PROMPT | Prompt 过长，各领域精度下降 | 高 |
| 3 | 工具调用和闲聊走同一分支 | 闲聊场景浪费了工具列表 token | 中 |
| 4 | 多个独立查询串行执行 | "今天收入+天气"需串行两次 LLM | 中 |
| 5 | 子智能体之间无协作 | "超时了怎么办"需要知识库+订单联动 | 中 |
| 6 | 新增业务需修改主 Agent | 违反开闭原则 | 低 |

### 2.2 现有工具分布

| 工具 | 当前归属 | 业务领域 | 数据依赖 |
|------|---------|---------|---------|
| `query_order` | riderAgent | 配送履约 | 订单数据库 |
| `query_income` | riderAgent | 财务结算 | 收入数据库 |
| `penalty_appeal` | riderAgent | 财务结算（罚单） | 罚单数据库 |
| `search_knowledge` | riderAgent | 知识库 FAQ | ChromaDB |
| `query_weather` | riderAgent | 配送环境 | 第三方天气 API |
| `transfer_to_human` | riderAgent | 客服转接 | 工单系统 |

## 三、业务领域划分

### 3.1 划分原则

按以下维度对业务进行子智能体划分：

1. **业务独立性**：不同领域的工具和 Prompt 之间耦合度低
2. **意图互斥性**：同一用户消息通常只命中一个或少数几个领域
3. **数据亲和性**：同一领域的工具通常访问相同的数据源
4. **演进方向一致**：同一领域的未来扩展方向相近

### 3.2 四大业务领域

```
┌─────────────────────────────────────────────────────────────────┐
│                      骑手智能客服系统                            │
│                                                                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐           │
│  │ 订单履约  │ │ 财务结算  │ │ 知识问答  │ │ 配送环境  │           │
│  │          │ │          │ │          │ │          │           │
│  │ 订单查询  │ │ 收入查询  │ │ FAQ搜索  │ │ 天气查询  │           │
│  │ 配送进度  │ │ 提现记录  │ │ 规则咨询  │ │ 配送建议  │           │
│  │ 取餐码   │ │ 罚单查询  │ │ 流程指引  │ │ 路况预警  │           │
│  │ 异常上报  │ │ 罚单申诉  │ │ 政策解读  │ │ 加价说明  │           │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘           │
│                                                                 │
│  ┌──────────────────────────────────────────────────┐           │
│  │                  主控 Agent                       │           │
│  │     意图识别 → 路由分发 → 结果汇总 → 回复生成      │           │
│  └──────────────────────────────────────────────────┘           │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 各领域与工具的映射关系

| 领域 | 子智能体 | 当前工具 | 未来可扩展工具 |
|------|---------|---------|--------------|
| 订单履约 | OrderAgent | `query_order` | `report_incident`、`query_navigation` |
| 财务结算 | FinanceAgent | `query_income`、`penalty_appeal` | `query_balance`、`request_withdrawal` |
| 知识问答 | KnowledgeAgent | `search_knowledge` | `submit_feedback`、`query_policy` |
| 配送环境 | WeatherAgent | `query_weather` | `query_traffic`、`query_area_heat` |

**不归属任何子智能体的能力**：

| 能力 | 归属 | 说明 |
|------|------|------|
| `transfer_to_human` | 主控 Agent | 转人工是全局能力，任何子智能体都可以触发 |
| 闲聊/问候 | 主控 Agent | 不需要工具调用，直接生成回复 |

## 四、各子智能体详细设计

### 4.1 主控 Agent（Orchestrator）

#### 4.1.1 职责定义

主控 Agent 是整个系统的**入口和调度中心**，不直接调用业务工具，只负责：

1. **意图识别**：分析用户消息，判断需要哪些子智能体参与
2. **路由分发**：将消息（或消息片段）分发给对应的子智能体
3. **并行调度**：多个独立意图并行调用子智能体
4. **结果汇总**：将各子智能体的回复合并为最终回复
5. **闲聊处理**：识别问候、感谢、闲聊等非业务意图，直接回复
6. **转人工判断**：识别需要转人工的信号，触发转接流程

#### 4.1.2 意图分类体系

```typescript
type AgentType = 'order' | 'finance' | 'knowledge' | 'weather' | 'chat';

interface IntentResult {
  intents: Array<{
    agent: AgentType;
    subMessage: string;  // 分发给子智能体的消息片段
  }>;
  transferToHuman: boolean;
  transferReason?: string;
}
```

**分类规则**：

| 用户消息示例 | 识别结果 | 说明 |
|-------------|---------|------|
| "你好" | `[chat]` | 纯闲聊 |
| "谢谢" | `[chat]` | 纯闲聊 |
| "今天几单了" | `[order]` | 单意图 |
| "今天赚了多少钱" | `[finance]` | 单意图 |
| "怎么申诉罚单" | `[knowledge]` | 知识问答（问流程） |
| "帮我看下罚单P001" | `[finance]` | 罚单查询（需要调工具） |
| "外面下雨吗" | `[weather]` | 单意图 |
| "今天收入多少，天气怎么样" | `[finance, weather]` | 多意图并行 |
| "超时了会被罚款吗" | `[knowledge]` | 知识问答（规则咨询） |
| "你们这破系统，我要投诉" | `[chat]` + 转人工 | 情绪触发转人工 |
| "我要转人工" | 转人工 | 显式请求 |

#### 4.1.3 路由分发 Prompt

```
你是一个意图路由器。分析用户消息，判断需要哪个业务智能体来处理。

可选业务智能体：
- order：订单相关（查询订单、配送进度、取餐码、异常上报）
- finance：财务相关（收入查询、提现、罚单查询、罚单申诉）
- knowledge：知识问答（配送规则、平台政策、操作流程、FAQ）
- weather：天气相关（天气查询、配送环境、是否适合跑单）
- chat：非业务对话（问候、感谢、闲聊）

判断规则：
1. 如果用户问罚单"怎么申诉"/"申诉流程"，归类为 knowledge（问流程）
2. 如果用户要"查罚单"/"看罚单P001"，归类为 finance（查数据）
3. 如果消息同时涉及多个领域，拆分为多个意图
4. 如果是问候/感谢/闲聊，归类为 chat
5. 如果用户明确要求转人工或情绪激动，设置 transferToHuman 为 true

请以 JSON 格式返回，不要输出其他内容：
{"intents":[{"agent":"xxx","subMessage":"分发给该智能体的消息"}],"transferToHuman":false}

用户消息：{userMessage}
```

#### 4.1.4 多意图并行调度

```
用户："今天赚了多少？外面什么天气？"
          │
    ┌─────┴──────┐
    ▼             ▼
FinanceAgent   WeatherAgent    ← Promise.all 并行
    │             │
    ▼             ▼
"今日收入156.5"  "晴天22°C"
          │
          ▼
    主控汇总回复
"今天你跑了12单，赚了156.5元💰。外面是晴天22°C，挺适合跑单的！"
```

#### 4.1.5 结果汇总策略

```typescript
const mergeReplies = (
  results: Array<{ agent: AgentType; reply: string }>
): string => {
  if (results.length === 1) return results[0].reply;

  // 多个子智能体结果按领域顺序拼接，用过渡句连接
  const reply = results.map((r, i) => {
    if (i === 0) return r.reply;
    // 根据领域添加过渡
    const transitions: Record<string, string> = {
      finance: '另外，',
      weather: '再说天气方面，',
      order: '关于订单方面，',
      knowledge: '还有个相关信息，',
    };
    return transitions[r.agent] + r.reply;
  }).join('');

  return reply;
};
```

### 4.2 订单智能体（OrderAgent）

#### 4.2.1 业务背景

骑手的核心工作就是配送，订单相关查询是最高频的场景。骑手需要随时了解：
- 当前有多少订单在配送中
- 订单的实时状态（待取餐、配送中、已送达）
- 今日完成单量
- 异常订单（超时、商家出餐慢等）

#### 4.2.2 职责边界

| 负责 | 不负责 |
|------|--------|
| 订单状态查询 | 收入金额（→ FinanceAgent） |
| 配送进度查询 | 天气影响（→ WeatherAgent） |
| 取餐码/验证码 | 罚单规则（→ KnowledgeAgent） |
| 异常订单上报 | 申诉流程（→ KnowledgeAgent） |
| 历史订单查询 | — |

#### 4.2.3 工具清单

| 工具 | 输入 | 输出 | 状态 |
|------|------|------|------|
| `query_order` | `riderId`, `status?` | 订单列表、数量统计 | 已实现（模拟数据） |

#### 4.2.4 意图识别规则

| 意图 | 触发关键词/模式 | 示例 |
|------|--------------|------|
| 查询全部订单 | "订单"、"今天几单" | "今天跑了多少单" |
| 查询配送中 | "配送中"、"在送的"、"正在送" | "我现在有几个在送" |
| 查询已完成 | "完成了"、"已送达"、"送完了" | "送完了几个" |
| 查询异常订单 | "超时"、"异常"、"取消" | "有没有超时的单子" |
| 查询取餐码 | "取餐码"、"验证码" | "这个单的取餐码是多少" |

#### 4.2.5 System Prompt

```
你是一个外卖骑手订单查询助手。

你的职责：
1. 查询骑手的订单状态和配送进度
2. 告知骑手当前配送中的订单和已完成的订单
3. 对异常订单给出简要说明

回复风格：
- 简洁明了，突出关键信息（订单号、状态、时间）
- 用数字说话，不要废话
- 配送中的订单要强调 urgency

回复格式参考：
"你今天总共28单，完成了25单🚀
配送中的有2单：
• 订单#12345 配送中（已用时18分钟）
• 订单#12346 待取货（商家出餐中）
取消了1单（商家取消）"
```

#### 4.2.6 典型对话场景

```
骑手：今天几单了？
Agent：你今天完成了25单，还有2单在配送中，1单被取消了。继续加油💪

骑手：那个12345到哪了？
Agent：订单#12345正在配送中，已用时18分钟，请尽快送达⏰

骑手：有没有超时的？
Agent：目前没有超时订单，2单配送中都在正常时效内✅

骑手：取餐码是多少？
Agent：请问是哪个订单的取餐码？可以告诉我订单号吗？

骑手：12345
Agent：订单#12345的取餐码是 8826，到店后出示给商家即可。
```

#### 4.2.7 未来扩展方向

| 方向 | 说明 | 优先级 |
|------|------|--------|
| 实时位置追踪 | 接入地图服务，显示骑手到目的地的距离和时间 | 中 |
| 异常自动上报 | 超时/异常自动生成工单 | 低 |
| 订单取消申诉 | 骑手端取消订单的流程 | 低 |

### 4.3 财务智能体（FinanceAgent）

#### 4.3.1 业务背景

收入和罚款是骑手最关心的财务问题。骑手需要：
- 随时了解今日/本周/本月收入
- 查询提现状态和可提现金额
- 查看罚单记录
- 提交罚单申诉

#### 4.3.2 职责边界

| 负责 | 不负责 |
|------|--------|
| 收入查询 | 配送单量（→ OrderAgent） |
| 提现操作 | 天气补贴（→ KnowledgeAgent） |
| 罚单查询 | 申诉流程指导（→ KnowledgeAgent） |
| 罚单申诉提交 | 转人工客服（→ 主控 Agent） |
| 余额/账单查询 | — |

#### 4.3.3 工具清单

| 工具 | 输入 | 输出 | 状态 |
|------|------|------|------|
| `query_income` | `riderId`, `period?` | 收入明细、订单数、奖励 | 已实现（模拟数据） |
| `penalty_appeal` | `riderId`, `action`, `penaltyId?`, `reason?` | 罚单列表 / 申诉结果 | 已实现（模拟数据） |

#### 4.3.4 意图识别规则

| 意图 | 触发关键词/模式 | 示例 |
|------|--------------|------|
| 查询今日收入 | "今天赚了"、"今天收入" | "今天赚了多少" |
| 查询周期收入 | "本周"、"本月"、"上周" | "这周收入怎么样" |
| 查询可提现 | "提现"、"能提多少" | "我可以提多少钱" |
| 查询罚单 | "罚单"、"扣款"、"罚款" | "我被罚了多少钱" |
| 申诉罚单 | "申诉"、"不合理的罚单" + 罚单ID | "P001这个罚单我要申诉" |
| 查询账单明细 | "账单"、"明细" | "看下我的账单" |

#### 4.3.5 知识协同场景

财务智能体与知识智能体存在协作关系：

```
骑手："罚单怎么申诉？"

主控路由 → KnowledgeAgent（问流程，不查数据）
KnowledgeAgent："申诉流程：1. 进入罚单记录 2. 选择罚单 3. 填写理由 4. 提交"

---

骑手："帮我看下罚单P001，我要申诉"

主控路由 → FinanceAgent（查数据）
FinanceAgent："罚单P001：超时未送达，扣款20元，3月23日。请提供申诉理由。"
骑手："当时商家出餐晚了15分钟"
FinanceAgent：调用 penalty_appeal(action='appeal', penaltyId='P001', reason='...')
FinanceAgent："已提交申诉，申诉编号A1234，预计3个工作日反馈。"
```

#### 4.3.6 System Prompt

```
你是一个外卖骑手财务助手。

你的职责：
1. 查询骑手的收入信息（今日/本周/本月）
2. 查询可提现金额
3. 查询罚单记录和详情
4. 受理罚单申诉（需要罚单ID和申诉理由）

回复风格：
- 涉及金额时精确到分，格式规范
- 罚单信息要包含：罚单ID、原因、金额、日期、状态
- 申诉时要确认必要信息（罚单ID、理由），缺了要追问
- 不要主动推荐提现操作

重要：
- 申诉罚单必须同时提供罚单ID和申诉理由，缺少任何一项要追问
- 如果用户问"怎么申诉"（流程问题），建议咨询知识库
- 如果用户情绪激动，建议转人工客服
```

#### 4.3.7 典型对话场景

```
骑手：今天赚了多少？
Agent：今天你完成了12单，收入156.5元，其中配送费141.5元，奖励15元💰

骑手：可以提现吗？
Agent：你目前可提现856.5元，累计收入12580元。

骑手：我被罚了多少？
Agent：你有2条罚单记录：
• P001 超时未送达 -20元（3/23，待处理）
• P002 配送路线偏离 -10元（3/20，已申诉）

骑手：P001要申诉，商家出餐晚了15分钟
Agent：已为你提交罚单P001的申诉。
申诉编号：A1234
申诉理由：商家出餐晚了15分钟
预计3个工作日内反馈结果。

骑手：怎么申诉？
Agent：申诉需要提供罚单ID和申诉理由。你可以先查看罚单列表（"查罚单"），然后告诉我具体要申诉哪条，我来帮你提交。
```

### 4.4 知识智能体（KnowledgeAgent）

#### 4.4.1 业务背景

骑手经常需要查询平台规则、操作流程、政策解读等非实时数据。这类问题不需要调用业务系统 API，而是从知识库（RAG 向量搜索）中检索相关文档。

#### 4.4.2 职责边界

| 负责 | 不负责 |
|------|--------|
| 配送规则查询 | 实时订单数据（→ OrderAgent） |
| 申诉流程指引 | 实际申诉操作（→ FinanceAgent） |
| 平台政策解读 | 天气信息（→ WeatherAgent） |
| 账号问题 | 收入查询（→ FinanceAgent） |
| 操作指引 | — |
| FAQ 问答 | — |

#### 4.4.3 工具清单

| 工具 | 输入 | 输出 | 状态 |
|------|------|------|------|
| `search_knowledge` | `query`, `category?` | 匹配的知识条目列表 | 已实现（ChromaDB + 降级模拟） |

#### 4.4.4 意图识别规则

| 意图 | 触发关键词/模式 | 示例 |
|------|--------------|------|
| 配送规则 | "规则"、"规定"、"多久" | "配送超时怎么算" |
| 申诉流程 | "怎么申诉"、"申诉流程" | "罚单怎么申诉" |
| 账号问题 | "封号"、"认证"、"换手机" | "账号被限制了怎么办" |
| 收入规则 | "怎么算钱"、"单价" | "配送费怎么算的" |
| 等级体系 | "等级"、"升级" | "怎么提升等级" |
| 安全合规 | "保险"、"违章" | "配送出事故怎么办" |
| 操作指引 | "怎么用"、"如何操作" | "健康证怎么上传" |

#### 4.4.5 与 RAG 的深度集成

知识智能体是 RAG 能力的主要消费者。当前 `search_knowledge` 已接入 ChromaDB 向量搜索，知识智能体需要做到：

```
用户提问
  │
  ├─ search_knowledge(query) ← 向量检索相关文档
  │
  ├─ 检索到结果 → LLM 基于检索结果生成回答（引用来源）
  │
  └─ 未检索到结果 → LLM 基于通用知识回答（标注"以下为通用建议"）
```

#### 4.4.6 System Prompt

```
你是一个外卖骑手知识问答助手。

你的职责：
1. 从知识库中搜索骑手关心的问题（配送规则、平台政策、操作流程等）
2. 用简洁的语言解释规则和流程
3. 给出具体的操作步骤

回复风格：
- 先给出结论，再展开说明
- 涉及步骤时用编号列表
- 如果搜索到知识库内容，基于搜索结果回答
- 如果搜索不到相关内容，基于通用外卖配送知识回答，并标注"以下为通用建议，具体以平台规则为准"

重要规则：
- 不要编造不存在的规则或政策
- 涉及金额、时限等具体数字时，要强调"以实际为准"
- 如果用户的问题属于实时查询（如"我赚了多少"），引导用户直接问收入问题
```

#### 4.4.7 典型对话场景

```
骑手：配送超时会罚款吗？
Agent：会。普通订单配送时间为30分钟，恶劣天气可延长15分钟。超时后可能面临扣款处理，具体金额以平台规则为准。

骑手：罚单怎么申诉？
Agent：申诉流程：
1. 进入"我的"→"罚单记录"
2. 选择需要申诉的罚单
3. 点击"申诉"，填写申诉理由和相关证据
4. 提交后等待审核（通常3个工作日内反馈）
如果需要我帮你直接提交申诉，请告诉我罚单ID和申诉理由。

骑手：配送费怎么算的？
Agent：配送费根据距离、时段、天气等因素动态计算。高峰时段（11:00-13:00、17:00-19:00）会有额外补贴，恶劣天气也会加价。具体金额以订单详情页为准。
以下为通用建议，具体以平台规则为准。

骑手：健康证怎么上传？
Agent：知识库中暂未找到相关信息。通常操作路径为：
1. 进入"我的"→"资质认证"
2. 选择"健康证上传"
3. 拍照上传健康证正面照
4. 等待审核（1-3个工作日）
建议在APP中查看最新的操作指引。
```

### 4.5 天气智能体（WeatherAgent）

#### 4.5.1 业务背景

天气直接影响骑手的配送安全和收入。骑手需要：
- 出门前了解当天天气，决定是否出车
- 恶劣天气时了解加价政策和配送时效延长
- 下雨/高温等极端天气时的安全提醒

#### 4.5.2 职责边界

| 负责 | 不负责 |
|------|--------|
| 实时天气查询 | 收入影响（→ FinanceAgent） |
| 配送建议 | 配送规则（→ KnowledgeAgent） |
| 天气预警提示 | 实际加价政策（→ KnowledgeAgent） |
| — | 路况查询（未来扩展） |

#### 4.5.3 工具清单

| 工具 | 输入 | 输出 | 状态 |
|------|------|------|------|
| `query_weather` | `latitude?`, `longitude?` | 天气、温度、风力、湿度、紫外线 | 已实现（wttr.in API） |

#### 4.5.4 意图识别规则

| 意图 | 触发关键词/模式 | 示例 |
|------|--------------|------|
| 查询天气 | "天气"、"几度"、"下雨" | "今天天气怎么样" |
| 配送建议 | "适合跑单吗"、"能不能送" | "今天适合跑单吗" |
| 天气预警 | "暴雨"、"台风"、"高温预警" | "外面暴雨能送吗" |
| 温度查询 | "多少度"、"冷不冷"、"热不热" | "现在多少度" |

#### 4.5.5 System Prompt

```
你是一个外卖骑手天气助手。

你的职责：
1. 查询骑手当前位置的实时天气
2. 根据天气情况给出配送建议
3. 恶劣天气时提供安全提醒

回复风格：
- 先报核心信息（天气、温度），再给建议
- 简洁明了，骑手需要快速决策
- 恶劣天气要明确提示风险

配送建议参考：
- 晴天/多云：适合跑单，注意防晒
- 小雨：可以跑单，注意路滑，配送时间会延长
- 大雨/暴雨：不建议跑单，安全第一
- 高温（>35°C）：注意防暑，多补充水分
- 大风（>6级）：注意骑行安全，避免空旷路段
- 雾霾：建议佩戴口罩，减速慢行

重要：
- 天气数据来自第三方API，可能存在延迟
- 配送规则（如加价政策）请咨询知识库，不要自行编造
```

#### 4.5.6 典型对话场景

```
骑手：今天天气怎么样？
Agent：当前天气：晴 22°C，体感20°C
东南风12km/h，湿度45%
今天天气不错，适合跑单💪

骑手：外面下雨吗？
Agent：当前没有下雨，但今天下午3点后有70%的概率转小雨🌧️
建议上午多跑几单，下午带好雨衣。

骑手：今天能跑吗？
Agent：当前晴天22°C，很适合跑单！
不过下午3点后可能有小雨，建议合理安排接单节奏。

骑手：台风来了怎么办？
Agent：⚠️ 台风天气安全提醒：
1. 立即停止配送，寻找安全地点避风
2. 不要在空旷路段、桥梁、树下停留
3. 关注平台通知，台风期间会有停单措施
4. 确保手机充满电，保持通讯畅通
安全第一！等台风过后再出车。
```

## 五、跨智能体协作场景

### 5.1 多意图并行

最常见的跨智能体场景是用户一句话包含多个意图：

```
骑手："今天赚了多少？天气怎么样？"

主控识别：[finance("今天赚了多少"), weather("天气怎么样")]

并行执行：
  FinanceAgent → "今天12单，收入156.5元"
  WeatherAgent → "晴天22°C，适合跑单"

主控汇总："今天你跑了12单，收入156.5元💰。外面晴天22°C，适合跑单💪"
```

### 5.2 知识+工具协作

某些问题需要先查知识库确认规则，再调工具查数据：

```
骑手："超时了会罚多少钱？P001超时了"

主控识别：
  - "超时了会罚多少钱" → knowledge
  - "P001超时了" → finance

并行执行：
  KnowledgeAgent → "超时配送通常扣10-20元..."
  FinanceAgent → "P001已扣款20元，状态待处理"

主控汇总：融合两个结果回复
```

### 5.3 子智能体触发转人工

当子智能体识别到超出自身处理范围的情况时，返回特殊标记，由主控 Agent 处理：

```typescript
interface SubAgentResult {
  reply: string;
  transferToHuman?: boolean;  // 子智能体建议转人工
  transferReason?: string;
}
```

触发场景：
- FinanceAgent：申诉被拒绝，骑手情绪激动
- KnowledgeAgent：知识库中无相关信息且通用知识也无法覆盖
- OrderAgent：订单出现严重异常（事故、丢失等）

## 六、数据流与接口设计

### 6.1 主控与子智能体通信接口

```typescript
/**
 * 子智能体上下文（主控传递给子智能体）
 */
interface AgentContext {
  riderId: string;
  sessionId: string;
  location?: { latitude: number; longitude: number };
  history: Array<{ role: string; content: string }>;
}

/**
 * 子智能体接口定义
 */
interface SubAgent {
  name: AgentType;
  description: string;

  /** 同步处理，返回完整回复 */
  process(message: string, context: AgentContext): Promise<SubAgentResult>;

  /** 流式处理，逐 chunk yield */
  stream(message: string, context: AgentContext): AsyncGenerator<string>;
}

/**
 * 子智能体返回结果
 */
interface SubAgentResult {
  reply: string;
  transferToHuman?: boolean;
  transferReason?: string;
}
```

### 6.2 子智能体注册表

```typescript
const agentRegistry: Record<AgentType, SubAgent> = {
  order: new OrderAgent(),
  finance: new FinanceAgent(),
  knowledge: new KnowledgeAgent(),
  weather: new WeatherAgent(),
};
```

## 七、降级与容错策略

### 7.1 子智能体调用失败

```typescript
const invokeWithTimeout = async (
  agent: SubAgent,
  message: string,
  context: AgentContext,
  timeout: number
): Promise<SubAgentResult> => {
  try {
    return await Promise.race([
      agent.process(message, context),
      new Promise((_, reject) =>
        setTimeout(() => reject(new Error('timeout')), timeout)
      ),
    ]);
  } catch (error) {
    // 降级：返回错误提示，不阻断其他子智能体
    return {
      reply: `抱歉，查询【${agent.description}】时出了点问题，请稍后再试。`,
    };
  }
};
```

### 7.2 意图识别失败的降级

如果主控 LLM 返回的意图 JSON 解析失败或格式异常：

1. 降级到现有单体 Agent 的 `selectTool` 逻辑
2. 记录失败日志用于分析
3. 仍然保证正常回复，不中断用户体验

### 7.3 流式场景的容错

多意图并行场景下，如果其中一个子智能体的流式输出中断：

1. 已输出的内容保留
2. 中断的子智能体部分标注"查询失败，请稍后重试"
3. 其他子智能体正常输出不受影响

## 八、性能影响分析

### 8.1 LLM 调用次数对比

| 场景 | 当前架构 | 子智能体架构 | 说明 |
|------|---------|------------|------|
| 单意图+工具 | 2次（选择+生成） | 2次（路由+子Agent） | 相同 |
| 单意图+闲聊 | 2次 | 2次（路由+闲聊） | 相同 |
| 多意图（2个） | 串行4次 | 1次路由+2次并行子Agent | **减少1次** |
| 多意图（3个） | 串行6次 | 1次路由+3次并行子Agent | **减少2次** |
| 闲聊 | 2次（选择NO_TOOL+生成） | 2次（路由识别+闲聊） | 相同 |

### 8.2 Token 消耗对比

| 场景 | 当前 SYSTEM_PROMPT | 子智能体 Prompt | 差异 |
|------|-------------------|----------------|------|
| 订单查询 | 通用（覆盖6个领域） | 订单专用 | Token 减少 ~30% |
| 闲聊 | 通用 | 主控闲聊 | Token 减少 ~50% |
| 多意图 | 同上×N | 各子Agent 专用 | 总 Token 减少 ~40% |
